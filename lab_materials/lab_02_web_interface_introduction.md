# Лабораторная работа (вводный материал для Moodle)

## 1. Название и контекст

**Тема:** разработка учебного web‑интерфейса на Rust (Yew) и подключение к backend‑API.

Проект: `rust-gigachat-webapp` — клиент к API проекта `rust-gigachat-app`.
UI и backend запускаются как **два отдельных процесса** на разных портах.

## 2. Цель

Сформировать базовое понимание архитектуры «frontend ↔ backend» в учебном проекте
и подготовить студента к выполнению лабораторной работы по интеграции UI с API.

## 3. Что будет выполнено в лабораторной

- запуск backend‑API и UI как двух отдельных процессов;
- проверка доступности API из интерфейса;
- отправка запроса и получение ответа в UI;
- диагностика и устранение проблемы CORS (если проявляется);
- закрепление базовых навыков работы с Git и beads.

## 5. Рекомендуемый порядок изучения материалов

1. `docs/build_and_run.md` — пошаговый запуск UI и backend.
2. `docs/api_contract.md` — формат запросов и ответов.
3. `docs/common_issues.md` — частые ошибки (включая CORS).
4. `docs/yew_ui_guide.md` — основы компонентов и состояния в Yew.
5. `docs/ui_ux_requirements.md` — требования к интерфейсу.
6. `lab_materials/lab_work.md` — основной текст лабораторной работы.
7. `lab_materials/git_github_setup.md` и `lab_materials/git_version_control.md` — базовый workflow Git.
8. `lab_materials/beads_guide.md` — работа с задачами в beads.

## 6. Минимальные требования к окружению

- установлен Rust и цель `wasm32-unknown-unknown`;
- установлен Trunk;
- доступен Git;
- backend запущен локально (обычно `http://127.0.0.1:8000`).

## 7. Краткое напоминание по CORS

UI работает на одном порту (например, `8080`), а API — на другом (`8000`).
Это разные **origin**, поэтому браузер может блокировать запросы.
Решения описаны в `lab_materials/lab_work.md` и `docs/common_issues.md`.

## 8. Организационные требования

- решения фиксируются через Git (осмысленные коммиты);
- задачи сопровождаются beads (create → in_progress → close);
- секреты (токены, `.env`, приватные ключи) не попадают в репозиторий.
