# Практикум: система контроля версий Git

## 1. Цель

Сформировать устойчивые навыки работы с Git на уровне базового
профессионального workflow: фиксирование изменений, управление ветками,
синхронизация с удаленным репозиторием и корректное разрешение конфликтов.

## 2. Ключевые понятия

- **Репозиторий** — набор версий проекта, сохраненных в виде коммитов.
- **Рабочее дерево (working tree)** — текущие файлы проекта на диске.
- **Индекс (staging area)** — промежуточная область, где формируется снимок для коммита.
- **Коммит** — сохраненный снимок состояния проекта.
- **Ветка** — отдельная линия развития проекта (набор связанных коммитов).
- **Удаленный репозиторий (remote)** — удаленная копия проекта (например, на GitHub).

## 3. Базовый цикл работы

1. Проверить состояние:
   ```bash
   git status
   ```
2. Посмотреть изменения:
   ```bash
   git diff
   ```
3. Добавить изменения в индекс:
   ```bash
   git add <файл>
   # или
   git add .
   ```
4. Создать коммит:
   ```bash
   git commit -m "Краткое описание изменения"
   ```

### 3.1. Рекомендации к сообщениям коммитов

- Первая строка — короткое резюме (до 50 символов).
- Затем пустая строка.
- Далее — более подробное описание (если требуется).

Пример:
```text
Обновлены инструкции по запуску UI

Добавлено пояснение по двум серверам и базовому URL API.
```

## 4. Инициализация или клонирование

### 4.1. Создание локального репозитория
```bash
git init
```

### 4.2. Клонирование удаленного репозитория
```bash
git clone <URL>
```

## 5. Работа с ветками

### 5.1. Создание и переключение на ветку
```bash
git switch -c feature/lab-2
```

### 5.2. Просмотр веток
```bash
git branch
```

### 5.3. Слияние ветки
```bash
git switch main
git merge feature/lab-2
```

## 6. Синхронизация с удаленным репозиторием

### 6.1. Получение изменений без слияния
```bash
git fetch
```

### 6.2. Получение изменений со слиянием
```bash
git pull
```

### 6.3. Отправка изменений
```bash
git push origin main
```

Если репозиторий создавался локально, сначала добавьте remote:
```bash
git remote add origin <URL>
```

## 7. Игнорирование файлов

Для исключения временных и генерируемых файлов используется `.gitignore`:
```text
/target
/dist
*.log
.env
```

## 8. Просмотр истории

```bash
git log --oneline --decorate --graph
```

Для просмотра конкретного коммита:
```bash
git show <hash>
```

## 9. Типичные ошибки и их устранение

### 9.1. Забыли добавить файл в коммит
```bash
git add <файл>
git commit -m "Добавлен забытый файл"
```

### 9.2. Нужно удалить файл из индекса, но оставить его локально
```bash
git rm --cached <файл>
```

### 9.3. Конфликт при слиянии

1. Открыть файлы с маркерами конфликтов (`<<<<<<<`, `=======`, `>>>>>>>`).
2. Исправить вручную.
3. Добавить исправленный файл в индекс:
   ```bash
   git add <файл>
   ```
4. Завершить коммит:
   ```bash
   git commit
   ```

## 10. Минимальные требования к работе в Git

- Все изменения фиксируются через осмысленные коммиты.
- История коммитов логична и отражает ход решения задачи.
- Ветка `main` содержит только проверенные и завершенные изменения.
- Секретные данные (ключи, пароли) не попадают в репозиторий.
